# 后端创作链路改造说明

本文档用于说明后端接下来需要配合前端调整的内容，目标是让 Echo AI 的五种创作模式尽量对齐老师 `music-example` 中的创作流程。

## 1. 当前目标

前端创作页包含五种模式：

- 常规创作
- 梗歌制造机
- 情绪炼歌
- 看图写歌
- 为 TA 写歌

希望这五种模式都采用统一链路：

```text
前端输入创作素材
→ 后端 AI 生成歌词
→ 前端拿到 title / style / lyrics
→ 后端提交音乐生成任务
→ 前端轮询任务状态
→ 后端返回完整歌曲
→ 歌曲包含音频、本地封面、AI 乐评等信息
```

## 2. 现有后端接口情况

目前后端已经具备主要接口：

```text
POST /api/lyrics
生成歌词

POST /api/generate
提交 AI 音乐生成任务

GET /api/task/:id
查询任务状态

POST /api/song/:id/review
生成 AI 乐评

POST /api/dj/:id
生成 AI DJ 播报
```

也就是说，基础接口是有的，主要问题是：`/api/generate` 的任务流程还没有完全像老师 example 一样，把封面、乐评等内容一起串进任务里。

## 3. 需要重点修改的后端文件

主要关注这些文件：

```text
src/ai/ai-public.controller.ts
src/ai/ai-task.service.ts
src/ai/minimax.service.ts
src/ai/dto/lyrics-request.dto.ts
src/ai/dto/music-request.dto.ts
src/common/services/cover-storage.service.ts
src/common/services/audio-storage.service.ts
src/common/utils/song-mapper.ts
```

## 4. 歌词生成逻辑需要对齐

老师 example 中，五种模式都是通过同一个歌词接口处理，只是 `mode` 不同。

建议后端继续使用：

```text
POST /api/lyrics
```

请求参数建议统一为：

```json
{
  "mode": "song",
  "prompt": "夏天、校园、毕业季",
  "forWho": "朋友",
  "image": "base64 或图片数据"
}
```

字段说明：

```text
mode:
  song     常规创作
  meme     梗歌制造机
  emotion  情绪炼歌
  photo    看图写歌
  foryou   为 TA 写歌

prompt:
  用户输入的灵感、热梗、情绪、额外要求等

forWho:
  仅为 TA 写歌时使用

image:
  仅看图写歌时使用
```

注意：风格建议主要在前端选择，并在生成音乐阶段传入。歌词阶段可以不强依赖 `styles`，这样更接近老师 example 的逻辑。

## 5. 看图写歌的特殊问题

当前 `LyricsRequestDto` 中 `prompt` 可能有最小长度限制。如果看图写歌只上传图片、不填写额外要求，可能会因为 `prompt` 为空导致校验失败。

建议后端调整为：

```text
当 mode = photo 且 image 存在时，prompt 可以为空。
```

或者由后端自动补默认 prompt：

```text
请根据图片内容写一首歌
```

这样前端不需要强制用户再写灵感。

## 6. 音乐生成任务需要补完整

现在 `/api/generate` 的主要逻辑在：

```text
src/ai/ai-task.service.ts
processGenerateTask()
```

目前流程大致是：

```text
生成音乐
保存音频
创建 song
任务完成
```

建议改成：

```text
生成音乐
保存音频到 uploads/audio
创建 song
生成封面
保存封面到 uploads/covers
生成 AI 乐评
更新 song 的 coverImg / review
任务完成并返回完整 song
```

这样前端轮询 `/api/task/:id` 完成时，拿到的歌曲就已经包含：

```text
audioUrl
coverUrl
duration
aiReview
status
published
mode
```

前端就不需要在音乐生成完成后再单独调用封面和乐评接口，整体流程会更稳定。

## 7. 封面和音频必须本地持久化

MiniMax 返回的图片和音频 URL 可能有有效期，不能直接长期保存远程 URL。

后端需要确保：

```text
音频保存到 uploads/audio
封面保存到 uploads/covers
数据库只保存本地访问路径
```

例如：

```text
/audio/song_xxx.mp3
/covers/song_xxx.jpeg
```

不要长期保存类似下面这种临时链接：

```text
https://xxx.oss-cn-wulanchabu.aliyuncs.com/xxx?Expires=...
```

否则过期后前端就会出现图片或音频无法加载的问题。

## 8. 静态资源路径需要保证可访问

服务器需要能访问：

```text
http://服务器IP:3000/audio/xxx.mp3
http://服务器IP:3000/covers/xxx.jpeg
```

后端需要在 `main.ts` 中正确暴露静态目录，例如：

```text
uploads/audio  → /audio
uploads/covers → /covers
```

同时 Docker 部署时要注意 `uploads` 目录持久化，避免容器重建后文件丢失。

## 9. 任务返回结构建议

`GET /api/task/:id` 完成时建议返回：

```json
{
  "taskId": "xxx",
  "status": "done",
  "progress": 100,
  "stage": "生成完成",
  "song": {
    "id": "xxx",
    "title": "歌曲标题",
    "style": "流行",
    "lyrics": "...",
    "audioUrl": "/audio/xxx.mp3",
    "coverUrl": "/covers/xxx.jpeg",
    "duration": 158,
    "aiReview": "AI 乐评内容",
    "mode": "song",
    "status": "draft",
    "published": false
  }
}
```

这样前端只需要根据 `status === done` 获取 `song`，不需要再拼多个接口结果。

## 10. 建议优先级

建议后端按这个顺序处理：

```text
1. 确认 /api/lyrics 五种 mode 都能正常返回 title / style / lyrics
2. 调整 photo 模式 prompt 可为空的问题
3. 修改 /api/generate 的 processGenerateTask，让它自动生成并保存音频、封面、乐评
4. 确保 /audio 和 /covers 静态资源线上可访问
5. 确保 Docker 部署时 uploads 目录不会因容器重建丢失
6. 更新 Swagger 文档中的请求和返回示例
```

## 11. 前后端约定

后端完成后，前端会统一改成：

```text
AI 写词：POST /api/lyrics
生成音乐：POST /api/generate
轮询任务：GET /api/task/:id
```

五种创作模式不再各自写不同的生成逻辑，而是统一走任务链路。

这样后续维护会更清晰，也更接近老师 example 的实现方式。
