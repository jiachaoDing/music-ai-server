## 1. 引言

### 1.1 编写目的

本文档用于以软件需求规格说明书（Software Requirements Specification，SRS）的格式，明确 Echo AI 音乐社区的系统范围、用户角色、外部接口、功能需求、数据需求、非功能需求、优先级和验收标准。

本文档基于对线上运行的基础框架（`http://120.27.120.22:7777/` ）的完整功能分析，作为后续系统设计、数据库设计、接口设计、前端开发、后端开发、测试验收、部署联调和项目答辩的依据。

### 1.2 项目背景

Echo AI 音乐社区是一个面向普通用户和轻量创作者的 AI 音乐创作与社区互动平台。项目已有基础框架运行于 `http://120.27.120.22:7777/`， 该框架为单体 Web 应用，覆盖了完整的用户流程。

正式项目在 `music-ai-web + music-ai-server` 前后端分离架构下，以基础框架为功能参照，完成所有功能的复现、增强和数据持久化。基础框架中的全部功能统一标记为 P0 复现范围；搜索和筛选不在基础框架中，不作为当前 P0 交付。

## 2. 总体描述

### 2.1 产品前景

Echo AI 音乐社区通过 AI 生成能力降低音乐创作门槛，通过社区功能提升作品传播和互动。正式系统需要在 React + NestJS + PostgreSQL + Prisma + Swagger 架构下，复现基础框架中已有的全部产品能力，并实现数据持久化、接口文档化和可测试部署。

### 2.2 产品功能概述

系统主要功能包括：

1. **用户账号**：注册、登录、保持登录、注销登录；邀请码注册制；回声积分系统（获取、消耗、流水）。
2. **AI 创作**：5 种创作模式（常规创作、梗歌制造机、情绪炼歌、看图写歌、为TA写歌）；灵感关键词输入、风格选择、AI 歌词生成、歌词二次编辑、AI 音乐生成、AI 封面生成、生成状态展示、失败重试。
3. **AI 音乐制作人**：一句话主题 → AI 策划概念 EP，自动生成多首歌曲、专辑名称、简介和封面。
4. **作品管理**：作品预览、发布前编辑（歌名、歌词、简介、标签、封面）、发布到社区、私密/公开状态切换、草稿箱。
5. **社区互动**：首页三栏信息流（同频/推荐/热榜）、作品详情、播放、点赞、收藏、歌单、树洞留言（支持匿名）、热门推荐。
6. **AI 主理人**：主理人主页、今日翻牌精选、灵感推荐、风格推荐、AI 乐评、AI 发布文案、话题抛引、每日挑战、社区运营辅助。
7. **PK 擂台**：发起对决、两首歌曲 PK、社区投票、票数可视化。
8. **时运曲/日历打卡**：每日打卡测时运、连续签到徽章系统（3/7/15/30/100天）、能量日历、幸运色/数字、宜忌行动指引、治愈插画、能量曲线、时运曲生成（演唱版/纯音乐版）、时运卡海报分享。
9. **专辑与二创**：专辑创建与详情、翻唱/二创、翻唱进化树可视化。
10. **电台**：底部导航电台页、14 个心情场景主题、时间自适应推荐（此刻推荐）、主题定时轮换、纯音乐 BGM 生成。
11. **AI DJ 播报**：TTS 语音播报，为作品生成开场白/推荐语。
12. **海报分享**：歌曲海报（封面+歌词金句+标签+二维码）、时运卡海报、下载保存。
13. **播放系统**：迷你底部播放器、全屏沉浸播放页、音频可视化频谱、播放队列管理、循环模式（列表循环/单曲循环/顺序播放）。
14. **管理后台**：用户管理、作品管理、评论管理、邀请码管理、话题/挑战管理。
15. **版权保护**：创作者信息记录、创作来源记录、AI 生成声明、发布前确认、作品举报。

### 2.6 设计和实现约束

1. 正式项目必须基于 `music-ai-web + music-ai-server` 前后端分离架构实现。
2. 基础框架中已有功能应全部复现，统一归入 P0。
3. 前端不得保存 MiniMax API Key。
4. AI 调用必须由后端统一封装。
5. 用户密码必须加密存储，不允许明文保存。
6. `.env`、数据库密码、AI Key、服务器密码等敏感信息不得提交到代码仓库。
7. 作品生成后、发布前，音乐名字可以编辑。
8. 作品生成后默认为草稿/私密状态，仅作者可见。
9. AI 生成采用异步任务模式：提交 → 排队 → 执行 → 完成/失败，前端轮询状态。
10. 搜索、筛选因基础框架中不存在，不纳入 P0。
11. 移动端优先：页面最大宽度 480px，底部导航栏悬浮玻璃态设计。

### 3.3 通信接口

1. 前端通过 HTTP/HTTPS 调用后端 REST API。
2. 请求和响应数据格式为 JSON。
3. 登录接口返回 JWT Token（Bearer 认证）。
4. 需要登录的接口必须校验 Token 有效性；Token 过期返回 401，前端自动弹出登录门禁。
5. 管理接口必须校验管理员角色。
6. AI 接口由后端调用 MiniMax API（文本生成、音乐生成、封面生成、TTS），前端不得直接调用第三方 AI 服务。
7. 所有主要接口应在 Swagger（`/docs`）中可查看、可调试。
8. AI 音乐生成采用异步任务模式：POST 提交任务 → 返回 taskId → 前端轮询 GET `/api/task/:id` → 返回 status（queued/running/done/error）。
9. 文件（音频、封面图）通过后端静态文件服务提供，URL 格式为 `/audio/:id.mp3`、`/covers/:id.jpg`。

| published | 已发布到社区 | 公开可见 |
| private | 作者手动转为私密 | 仅作者可见 |
| removed | 管理员下架 | 不公开展示 |

| meme | 梗歌制造机 | 热梗/吐槽→洗脑神曲 |
| emotion | 情绪炼歌 | 心情/日记→专属歌曲 |

### 5.5 回声积分经济

| --- | --- | --- |
| 注册 | +N | 新用户注册赠送 |
| 每日打卡 | +N | 打卡奖励 |

| 作品被翻唱 | +N | 他人基于此作品二创 |

### 5.6 数据约束

1. 用户 name 必须唯一。
2. 用户密码只能保存加密摘要（bcrypt/argon2）。
3. 邀请码状态需记录创建、使用和失效。
4. 作品标题可由用户自行填写，也可由 AI 生成后自动回填；发布前作者可修改。
5. 作品生成后默认 published=false（草稿），仅作者本人可见。
6. 只有作者本人可编辑、发布或转私密自己的作品。
7. 已发布作品（published=true）进入社区公开流。
8. 同一用户对同一作品只能保留一个点赞关系（不可重复点赞，不可取消）。
9. 同一用户对同一作品可收藏到多个歌单。
10. 二创作品必须记录 originId（原作品来源）。
11. 同一用户对同一擂台只能投票一次。
12. 同一用户每天只能打卡一次（按 date 字段保证唯一）。
13. AI 主理人生成内容需记录类型、生成时间和关联对象。
14. 积分变动必须记录流水（PointsLedger），保证可追溯。
15. Token 过期后返回 401，前端清除登录态。

## 7. 优先级和交付计划

### 7.1 P0 核心交付（基础框架功能复现）

P0 必须完成基础框架中所有功能的复现和核心闭环：

1. **用户账号**：注册、登录、保持登录、注销登录、个人中心、邀请码、回声积分、积分流水。
2. **AI 创作（5 模式）**：常规创作、梗歌制造机、情绪炼歌、看图写歌、为TA写歌；灵感输入、风格选择、歌词生成、歌词编辑、音乐生成、封面生成、生成状态展示、失败重试。

4. **作品管理**：草稿箱、作品预览、发布前编辑（歌名/歌词/简介/标签/封面）、发布、转为私密、作品详情。

7. **PK 擂台**：擂台列表、发起对决、投票、票数可视化。
8. **时运曲/日历打卡**：每日打卡、连续签到徽章（5档）、月历视图、能量曲线、治愈插画、时运曲生成（演唱/纯音乐）、今日宜听、时运卡海报。

10. **电台**：底部导航电台页、14 心情场景主题、时间自适应推荐、定时轮换、纯音乐生成。
11. **AI DJ 播报**：TTS 播报生成、DJ→歌曲自动衔接。
12. **海报分享**：歌曲海报（封面+歌词金句+标签+二维码）、时运卡海报、下载。
13. **播放系统**：迷你播放器、沉浸全屏播放、音频可视化、播放队列、循环模式（3种）、进度拖动。
14. **管理后台**：用户管理、作品管理、评论管理、邀请码管理、话题/挑战管理。
15. **数据与技术**：P0 数据持久化（PostgreSQL+Prisma）、Swagger 接口文档、异步任务机制、主流程测试和部署验证。

### 9.6 前端技术要点

1. **页面路由**：SPA 单页路由，通过切换 `view` 容器 innerHTML 实现页面切换。
2. **动效系统**：页面进入淡入上滑动画（`View.animate`）、按钮涟漪效果（`ripple` span 动态注入）、播放状态脉冲。
3. **背景**：固定 3 个流体光斑（紫/粉/青），大模糊半径缓慢浮动。
4. **音频可视化**：Web Audio API 创建 AnalyserNode → `getByteFrequencyData` → Canvas 绘制圆形律动（迷你播放器）和全屏频谱条（沉浸页）。
5. **海报**：离屏 Canvas（640×900）→ 封面图/渐变+歌名+作者+歌词金句+标签+二维码 → `toDataURL` → 下载。
6. **二维码**：调用 `/api/qr?text=URL` 生成，作为海报右下角元素。
7. **异步轮询**：`submitGenerate` → 拿到 `taskId` → 每 2.5s 轮询 `/api/task/:id` → 根据 status 更新 UI → done 时展示结果。
8. **本地存储**：Token（`echo_token`）、用户信息（`echo_user`）、点赞记录（`liked`）、歌单（`playlists`）、打卡记录（`echo_checkins`）、循环模式（`repeatMode`）、PK 投票记录（`voted_*`）。
